Chapter 01 — Why OOP was needed
Problem: Data हरू Stack र Heap मा scattered थिए। Addresses मात्र थिए — कुन data कुन person को थाहा थिएन। कोही पनि access र modify गर्न सक्थ्यो।
// Before OOP — scattered, no authority String name = "rupesh"; // 0xc9 — Stack/Heap int age = 33; // 0xbc — Stack String addr = "nepal"; // 0xaa — Heap String phone= "xxxx-xxxx"; // 0x1a — Heap // ❌ No single unit. No authority over data. // ❌ Anyone can: name = "bhushan"; — mess up!
Problem 1 Scattered Memory
एउटै person को data (name, age, address, phone) Stack र Heap मा अलग-अलग addresses मा थियो। Single unit को रूपमा treat गर्न सकिँदैनथ्यो।
Problem 2 No Authority
Data independent थियो — कुनै पनि code ले access र modify गर्न सक्थ्यो। name = "bhushan" जहाँबाट पनि गर्न मिल्थ्यो। Ownership थिएन।
Problem 3 Manual Arguments
Method मा data pass गर्न: print(name, age, addr, phone) — सबै manually pass गर्नुपर्थ्यो। Error-prone र tedious।
Solution Class + Object
Real-world entities लाई represent गर्न Class (blueprint) र Object (instance) concept आयो। Related data एउटै unit मा, authority सहित।
// After OOP — single unit, authority, encapsulated class Person { String name; int age; String address; String phoneNo; } Person rupesh = new Person(); // ✓ One object — one unit Person bhushan= new Person(); // ✓ Separate object — no conflict
OOP को core idea: Real world मा student, person, employee जस्ता entities छन् — तिनलाई program मा represent गर्न Class (abstract blueprint) र Object (actual instance) concept आयो। Class = idea, Object = reality।

new vs constructor: new = घर बनाउने 🏠 (Heap मा memory allocate) | Constructor = घर बनाउँदा चल्ने process | this()/super() = त्यो process भित्र अर्को process call
Constructor Chaining — Teacher1 style (Best Practice)
"Next" थिचेर देख्नुस् कसरी Teacher(tid) → Teacher(tid,name) → ... → actual assignment सम्म chain हुन्छ।
// Best practice — bottom-up chaining (Teacher1 style) class Teacher { int tid; String name; String dept; int salary; Teacher(int tid) { this(tid, "UNKNOWN"); // ← delegates up } Teacher(int tid, String name) { this(tid, name, "UNKNOWN"); } Teacher(int tid, String name, String dept) { this(tid, name, dept, 0); } Teacher(int tid, String name, String dept, int salary) { this.tid = tid; // ← actual assignment एकै ठाउँमा this.name = name; this.dept = dept; this.salary = salary; } }

किन 8 को multiple? CPU ले memory बाट data 8 bytes को chunk मा पढ्छ। Object 8-byte aligned भयो भने एकै read मा data मिल्छ। नभए एउटै field दुइटा cache line मा पर्छ — slow।
Lock States (Mark Word)
Biased Lock
एउटै thread ले repeatedly access गर्छ। Fast — कुनै actual locking mechanism छैन।
Lightweight Lock
दुइटा threads compete गर्छन् — CAS (Compare and Swap) use हुन्छ। Stack मा lock record राखिन्छ।
Heavyweight Lock
High contention — OS level mutex। Thread block हुन्छ। Expensive।
Chapter 05 — String Pool & Heap
String Pool भनेको Heap भित्रको sub-region हो। String literals ("rupesh") यहाँ जान्छन् — interned। new String() ले Heap मा छुट्टै object बनाउँछ।

String Pool (Heap sub-region)

@SP001"rupesh"
@SP002"nepal"
@SP003"Name: "
@SP004"xxxx-xxxx"
Interned strings — shared, reused

Normal Heap

@H101new String("rupesh")
@H102"Name: rupesh" (concat result)
Runtime created — not interned
// literal → String Pool (@SP001) String a = "rupesh"; // a → @SP001 String b = "rupesh"; // b → @SP001 (same! reused) // new String() → Normal Heap (@H101) String c = new String("rupesh"); // c → @H101 (new object) // + "rupesh" also in Pool @SP001 a == b // true — same Pool address a == c // false — different addresses a.equals(c) // true — same content // ↑ त्यसैले String compare मा == होइन .equals() use गर्नुस्! // Concatenation String result = "Name: " + "rupesh"; // → "Name: " Pool मा, "rupesh" Pool मा // → Java 9+: invokedynamic + StringConcatFactory // → result "Name: rupesh" → Normal Heap (@H102) // → return गर्दा @H102 को address return हुन्छ, value होइन
Loop मा + operator: प्रत्येक concatenation मा नयाँ Heap object बन्छ — expensive! Loop भित्र StringBuilder use गर्नुस्।
